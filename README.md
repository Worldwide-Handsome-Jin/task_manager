# 📋 Task Manager API

> Полноценный REST API для управления задачами с JWT авторизацией, написанный на Python + FastAPI.  
> Каждый пользователь видит только свои задачи — данные изолированы и защищены токеном.

---

## 🧠 Что делает этот проект и зачем он нужен

Представь, что ты делаешь мобильное приложение или сайт со списком задач. Приложению нужен **бэкенд** — сервер, который хранит данные, проверяет кто ты такой и отдаёт правильные данные нужному пользователю.

Именно это и делает данный проект. Это **API (Application Programming Interface)** — набор HTTP эндпоинтов, к которым может обращаться любой клиент: мобильное приложение, веб-сайт, другой сервис или просто Postman.

**Что умеет этот API:**
- Регистрировать пользователей и выдавать JWT токены для авторизации
- Создавать, читать, обновлять и удалять задачи (CRUD)
- Фильтровать задачи по статусу и приоритету
- Гарантировать что пользователь А никогда не увидит задачи пользователя Б
- Валидировать все входящие данные и возвращать понятные ошибки

## 🛠 Технологический стек

| Технология | Зачем используется |
|---|---|
| **Python 3.12** | Язык программирования |
| **FastAPI** | Веб-фреймворк для создания API. Автоматически генерирует документацию и валидирует данные |
| **PostgreSQL** | Реляционная база данных для хранения пользователей и задач |
| **SQLAlchemy** | ORM (Object-Relational Mapper) — позволяет работать с БД через Python-объекты вместо SQL |
| **Pydantic** | Валидация данных. Если клиент передал кривые данные — Pydantic остановит их до базы |
| **JWT (JSON Web Token)** | Авторизация без сессий. Пользователь логинится → получает токен → предъявляет его при каждом запросе |
| **bcrypt** | Хэширование паролей. Пароли никогда не хранятся в открытом виде |
| **Alembic** | Версионированные миграции базы данных |
| **Docker + Docker Compose** | Контейнеризация. Весь проект запускается одной командой на любом компьютере |
| **pytest** | Автоматические тесты |

## 🚀 Быстрый старт

### Требования

- [Docker Desktop](https://www.docker.com/products/docker-desktop/) — единственное что нужно установить

### 1. Клонируй репозиторий

```bash
git clone https://github.com/твой-username/task_manager.git
cd task_manager
```

### 2. Создай файл с переменными окружения

Скопируй шаблон и при необходимости измени значения:

```bash
cp .env.example .env
```

Содержимое `.env`:

```env
DATABASE_URL=postgresql+asyncpg://taskuser:taskpass@db:5432/taskdb
SECRET_KEY=замени-на-свой-секретный-ключ-минимум-32-символа
ALGORITHM=HS256
ACCESS_TOKEN_EXPIRE_MINUTES=30
DEBUG=false
```

> ⚠️ **Важно:** Никогда не коммить `.env` в публичный репозиторий. Файл уже добавлен в `.gitignore`.

### 3. Запусти проект

```bash
docker compose up --build
```

При первом запуске Docker автоматически:
- Скачает образы PostgreSQL и Python
- Установит все зависимости
- Создаст таблицы в базе данных
- Запустит API сервер

### 4. Проверь что всё работает

Открой в браузере: **http://localhost:8000/docs**

Ты увидишь интерактивную документацию Swagger — можно тестировать API прямо из браузера.

Или проверь health-check эндпоинт:

```bash
curl http://localhost:8000/health
# {"status":"healthy","app":"Task Manager API"}
```

## 🔑 Как пользоваться API

### Шаг 1 — Регистрация

```bash
curl -X POST http://localhost:8000/auth/register \
  -H "Content-Type: application/json" \
  -d '{
    "email": "user@example.com",
    "username": "myuser",
    "password": "mypassword123"
  }'
```

**Ответ (201 Created):**
```json
{
  "id": 1,
  "email": "user@example.com",
  "username": "myuser",
  "is_active": true,
  "created_at": "2026-03-16T10:00:00Z"
}
```

### Шаг 2 — Логин и получение токена

```bash
curl -X POST "http://localhost:8000/auth/login?email=user@example.com&password=mypassword123"
```

**Ответ (200 OK):**
```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "bearer"
}
```

Сохрани `access_token` — он нужен для всех последующих запросов.

### Шаг 3 — Создание задачи

Передавай токен в заголовке `Authorization`:

```bash
curl -X POST http://localhost:8000/tasks/ \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Изучить FastAPI",
    "description": "Прочитать официальную документацию",
    "priority": "high",
    "status": "todo"
  }'
```

**Ответ (201 Created):**
```json
{
  "id": 1,
  "title": "Изучить FastAPI",
  "description": "Прочитать официальную документацию",
  "status": "todo",
  "priority": "high",
  "is_completed": false,
  "due_date": null,
  "created_at": "2026-03-16T10:05:00Z",
  "owner_id": 1
}
```

### Получение задач с фильтрацией

```bash
# Все задачи
curl http://localhost:8000/tasks/ \
  -H "Authorization: Bearer ваш_токен"

# Только задачи в процессе с высоким приоритетом
curl "http://localhost:8000/tasks/?status=in_progress&priority=high" \
  -H "Authorization: Bearer ваш_токен"

# Пагинация: вторая страница по 10 задач
curl "http://localhost:8000/tasks/?skip=10&limit=10" \
  -H "Authorization: Bearer ваш_токен"
```

### Обновление задачи

Можно обновить только нужные поля — остальные останутся без изменений:

```bash
curl -X PATCH http://localhost:8000/tasks/1 \
  -H "Authorization: Bearer ваш_токен" \
  -H "Content-Type: application/json" \
  -d '{"status": "done", "is_completed": true}'
```

## 🔒 Как работает авторизация

Проект использует **JWT (JSON Web Token)** — это стандартный способ авторизации в REST API.

**Процесс:**

1. Пользователь отправляет логин и пароль на `/auth/login`
2. Сервер проверяет пароль через bcrypt и создаёт JWT токен со сроком жизни 30 минут
3. Токен возвращается клиенту
4. Клиент сохраняет токен и добавляет его в заголовок `Authorization: Bearer <token>` при каждом запросе
5. Сервер расшифровывает токен, достаёт email пользователя и загружает его из БД

**Почему это безопасно:**
- Пароли хранятся только в виде bcrypt хэша — даже если БД утечёт, пароли не скомпрометированы
- Токен подписан секретным ключом (`SECRET_KEY`) — его нельзя подделать без ключа
- Токен имеет срок жизни — даже украденный токен перестанет работать

## ⚙️ Переменные окружения

| Переменная | Описание | Пример |
|---|---|---|
| `DATABASE_URL` | Строка подключения к PostgreSQL | `postgresql+asyncpg://user:pass@localhost:5432/db` |
| `SECRET_KEY` | Секретный ключ для подписи JWT токенов | Минимум 32 случайных символа |
| `ALGORITHM` | Алгоритм подписи JWT | `HS256` |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | Время жизни токена в минутах | `30` |
| `DEBUG` | Включить SQL логирование | `true` / `false` |

Сгенерировать надёжный `SECRET_KEY`:

```bash
python -c "import secrets; print(secrets.token_hex(32))"
```

## 🗄️ Миграции базы данных (Alembic)

Когда изменяешь модели — нужно создать миграцию, а не пересоздавать таблицы вручную:

```bash
# Создать миграцию после изменения моделей

alembic downgrade -1

# Посмотреть историю миграций
alembic history
```

## 🤝 Коды ответов

| Код | Значение | Когда возвращается |
|---|---|---|
| `200` | OK | Успешный GET или успешный логин |
| `201` | Created | Успешное создание ресурса |
| `204` | No Content | Успешное удаление |
| `400` | Bad Request | Нарушение бизнес-логики (например, email уже занят) |
| `401` | Unauthorized | Неверный или истёкший токен |
| `403` | Forbidden | Токен не передан |
| `404` | Not Found | Ресурс не найден или принадлежит другому пользователю |
| `422` | Unprocessable Entity | Ошибка валидации данных (Pydantic) |
| `500` | Internal Server Error | Ошибка на сервере |
